8.1 Prompt conditions
Four prompt conditions are compared:
Base zero-shot
Two-shot few-shot
Decomposed candidate-tracking
Expanded-definition
These conditions were chosen because they directly test four central heuristic questions:
Condition	Heuristic question
Base zero-shot	Is a minimal prompt already sufficient?
Two-shot few-shot	Do a small number of demonstrations help or hurt?
Decomposed candidate-tracking	Does structured reasoning help more than examples?
Expanded-definition	Is extra instruction worth the context cost?

8.2 Evaluation stages
The experiments proceed in four stages:
Dev stage: run all four prompt conditions on dev.csv
Challenge stage: run the best one or two prompt conditions on challenge.csv
Holdout stage: run exactly one frozen final prompt on holdout.csv
Qualitative review: inspect and categorize errors



10. Prompt Condition 1: Base Zero-Shot
10.1 Heuristic being tested
A compressed zero-shot prompt may be sufficient for pronoun resolution and may be the most efficient default.
10.2 Why this condition is included
This is the baseline condition. It tests the minimum viable prompt for the task.
If this condition already performs well, then longer or more elaborate prompts may not be justified.

12. Prompt Condition 3: Decomposed Candidate-Tracking
12.1 Heuristic being tested
Pronoun resolution may benefit from structured reasoning that explicitly directs the model to identify candidates and resolve the reference before choosing.
12.2 Why this condition is included
This condition tests whether task decomposition helps more than few-shot demonstration.
It is especially useful for checking whether the model benefits from a guided procedure rather than from analogical pattern-matching.

13. Prompt Condition 4: Expanded-Definition
13.1 Heuristic being tested
Adding a short linguistic definition and decision rule may improve performance, but may also cost tokens without adding much value.
13.2 Why this condition is included
This condition tests whether a small amount of extra explanation is useful enough to justify the added context.
It directly supports the broader heuristics question of instructional detail versus token economy.

16. Validation and Scoring
16.1 Primary metric
Accuracy
Primary task score:
accuracy on the parsed answer field
A prediction is correct if:
the output is valid JSON
the parsed value is exactly A or B
the parsed value matches the gold answer
16.2 Secondary metrics
Challenge-set accuracy
This measures whether a prompt that performed well on dev remains strong on more difficult items.
Holdout accuracy
This is the final untouched estimate after prompt selection is complete.
Malformed-output rate
The proportion of items for which the model fails to produce parseable output in the required schema.
Malformed includes:
invalid JSON
extra explanatory text if parser cannot accept it
missing answer key
answer not equal to A or B
Refusal / abstention rate
The proportion of items where the model refuses, declines, or refuses to choose.
Examples include:
“I cannot determine”
“insufficient information”
policy refusal
hedged answer without valid JSON
Token usage
Track:
input tokens
output tokens
total tokens
Where supported by the provider.
Correct-per-token efficiency
Optional but recommended:
number of correct predictions divided by total token usage for the condition
This helps compare longer and shorter prompts fairly.
16.3 Optional stability metric
If budget allows, rerun a fixed 20-item slice and compute:
agreement rate across repeated runs
This tests whether a prompt is stable or brittle.

20. Why These Validation Choices Were Used
This experiment uses multiple validation signals because a prompt is not “good” just because it gets a slightly better dev-set accuracy once.
A useful prompt for linguists should ideally be:
accurate
robust on harder examples
parseable without hand-cleaning
efficient enough to use at scale
interpretable enough to support error analysis
That is why the experiment tracks:
accuracy
challenge performance
holdout performance
malformed-output rate
refusal rate
token usage
qualitative failure modes
